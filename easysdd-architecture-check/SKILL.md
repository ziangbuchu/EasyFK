# easysdd-architecture-check

**架构一致性检查子工作流** —— 只做检查与建议，不做修改。

> "先把不和谐点说清楚，再决定改不改。"

---

## 一、适用场景

- 检查一个 design 文档内部是否自洽（术语、约束、契约、推进顺序是否互相矛盾）
- 检查一个 design 文档与当前代码实现是否一致（承诺了什么、代码是否真的做到）
- feature 接近实现或验收阶段，先做一次架构体检

不适用场景：

- 用户希望直接修复（应转到 `easysdd-feature-implement` 或 `easysdd-issue-fix`）
- 用户希望做拍板决策（应转到 `easysdd-decisions`）
- 用户还没有 design 文档（应先转到 `easysdd-feature-design`）

---

## 二、单目标规则（硬约束）

本技能**每次只检查一个目标**，目标二选一：

1. `design-internal`：仅检查 design 内部一致性 / 连贯性
2. `design-vs-code`：仅检查 design 与代码的一致性 / 连贯性

如果用户一次提出两个目标，必须先让用户选一个；另一个目标留到下一次检查。

---

## 三、涉及路径

本技能在正文里用自然语言术语引用路径（**方案 doc**、**架构中心目录**、**相关代码文件**），具体目录位置见主技能 `easysdd` 第二节"目录安排"。

---

## 四、工作流

### Phase 1：锁定检查目标

先确认：

- 当前检查目标：`design-internal` 或 `design-vs-code`
- 本次只检查哪个 feature（一个 feature 目录）
- 检查范围（例如只看第 0/2/3/5/6 节，或只看 auth 模块）

如果范围太大，要求收敛到一个可执行子范围。

### Phase 2：读取材料

共同必读：

- `AGENTS.md`
- 方案 doc 全文
- 架构中心目录相关文档（如 DESIGN.md）

额外读取：

- 若目标是 `design-vs-code`：必须读取与 design 第 2/3 节直接对应的代码文件

### Phase 3：执行检查（按目标模板）

#### 目标 A：design-internal

检查项（至少覆盖以下 6 类）：

1. 术语一致性：第 0 节术语是否在后续章节被同义词替换或语义漂移
2. 需求对齐：第 1 节摘要是否自洽、是否与用户已确认目标偏离
3. 契约闭环：第 2 节的契约示例是否在第 3 节有对应的改动计划和推进步骤
4. 示例与决策一致：第 2 节契约示例中的行为是否与第 1 节关键决策矛盾
5. 范围守护：第 3 节改动计划是否超出第 1 节的"明确不做"
6. 推进可执行性：第 3 节推进步骤是否可验证、依赖是否前后矛盾

#### 目标 B：design-vs-code

检查项（至少覆盖以下 6 类）：

1. 类型一致性：design 第 2 节契约示例中定义的核心类型/字段，代码中是否存在且语义一致
2. 行为一致性：design 第 2 节契约示例中声明的输入→输出，代码实际行为是否匹配
3. 写路径一致性：design 声明的写入口，代码是否出现额外旁路写入
4. 边界行为一致性：design 第 1 节异常/边界规则，代码是否实现
5. 改动边界一致性：design 第 3 节声明的改动范围，代码是否越界或漏实现
6. 推进结果一致性：design 第 3 节每步退出信号，对应代码状态是否可验证

### Phase 4：输出检查报告（仅报告，不修复）

输出格式：

```markdown
# 架构一致性检查报告

> 目标: design-internal | design-vs-code
> 范围: {feature}/{模块}/{章节范围}
> 日期: YYYY-MM-DD
> 结论: pass | pass-with-risk | fail

## 1. 检查摘要

一句话总结本次检查结论。

## 2. 不一致清单

| ID | 严重级别 | 位置 | 现象 | 影响 | 建议修复 |
|---|---|---|---|---|---|
| AC-01 | 高/中/低 | `{文件}:{行号}` 或 `design 第X节` | 描述不一致点 | 会造成什么后果 | 给出修复建议（不执行） |

## 3. 一致性良好项

列出 2-5 条检查通过的关键点，避免报告只剩负面信息。

## 4. 建议下一步

- 若为 fail：建议先修复 AC-01/AC-02，再重新运行本技能
- 若为 pass-with-risk：建议在实现/验收阶段重点回归哪些点
- 若为 pass：可进入下一阶段（如 implement 或 acceptance）
```

---

## 五、硬性边界（必须遵守）

1. **只检查，不修复**：禁止直接改 design、代码、配置
2. **单目标**：一次只能做 `design-internal` 或 `design-vs-code` 之一
3. **证据化**：每个不一致项都要有可定位位置（文件:行号 或 design 节）
4. **可执行建议**：建议必须具体到"改哪里、怎么改"，但不落盘改动
5. **不发散**：发现范围外问题只记为"观察项"，不展开深挖

---

## 六、退出条件

- [ ] 已明确并锁定单一检查目标
- [ ] 已完成对应目标的检查项覆盖
- [ ] 报告包含不一致清单 + 修复建议
- [ ] 报告未包含任何实际修复动作
- [ ] 用户确认本轮检查结论

---

## 七、反模式（看到就停）

- ❌ 一次同时做 design-internal 和 design-vs-code
- ❌ 发现问题后直接改代码或改文档
- ❌ 只说"这里不太对"，不给证据位置
- ❌ 建议过于抽象（例如"优化一下架构"）
- ❌ 从一个目标无限扩展到全仓库审计
