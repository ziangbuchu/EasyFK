# Rationale

## 如果能问用户，我会想确认的两件事（已用假设继续）

1. **"code review 类场景要不要进 bug-hunter？"** —— 比如用户说"这段代码你看看有没有 bug"（没有报错，只是怕有问题）。我假设 **不进**，因为你原话说能力范围是"从错误日志、stack trace、或异常症状里找根因"——这三种输入都要求"已经有故障信号"。如果实际上你也希望 review 类场景触发，需要把排除那句话放松。
2. **"没有错误日志但用户描述了异常症状"算不算？** —— 比如"我点按钮后没反应"。我假设 **算**，这属于"用户描述的异常症状"，你原话里第三类就是这个。description 里我用"异常表现、复现步骤"来涵盖。

---

## 旧 description 的问题

原文：`Helps find bugs in code`

按 §3.1 对照，这条同时踩了两个坑——既不触发又有乱触发风险，原因是：

- **"Helps ..." 是能力介绍，不是触发条件。** §5 第一条陷阱就是这个。Claude 读 description 时是在判断"用户当前问题该不该拉这个 skill"，它需要看到"用户想做 X 时用"，而不是"这个 skill 能做 X"。
- **"bugs in code" 语义极泛。** 写新代码、审查代码、重构、调试、防御性编程、单元测试——所有这些场景都能说自己在"处理代码里的 bug"。结果就是 Claude 在每种场景下都拿不准，干脆都不触发（这符合 Claude 默认少触发的倾向），所以用户只能 `/bug-hunter` 手动调。
- **没有用户原话。** §2.3 和 §3.2 都强调原话命中率比语义匹配高。原 description 里一个用户真实会说的短语都没有。
- **没有边界。** 既然能力范围只是"定位根因"（不包括"写代码时顺手避 bug"、不包括"code review"），就该写出来。不写，Claude 无从判断相邻场景要不要进。

---

## 改了什么

1. **把"Helps find bugs"换成触发条件描述**：`用户手里已经有一个"出问题的现象"……想找出根因并定位到具体出错代码时进入`。动词开头、指向用户状态，不指向 skill 能力。
2. **把你原话里的三种输入信号写死在 description 里**：报错日志、stack trace、异常表现、复现步骤。这是"有资格进 bug-hunter"的门槛——Claude 看到用户消息里有这类信号就知道是这个 skill，没有就知道不是。
3. **塞了 5 条用户原话触发语**。都是用户问 bug 时真实会说的中文短句，覆盖"这个报错怎么回事 / 看 stack trace / XX 坏了 / 这 bug 从哪来 / 排查异常"五种典型说法。
4. **加了明确排除**：`不用于写新代码时的顺手挑错或 code review`。这是 `bugs in code` 最容易被误匹配的相邻场景，必须点名。
5. **保留了"定位到具体代码行"这个你强调的交付形态**——让 Claude 知道这个 skill 不只是"说 bug 可能在哪"，是要钉到具体行。

## 预期的触发 / 不触发

### 应该命中（改完后）

- "这个报错怎么回事：`TypeError: cannot read property 'x' of undefined`" —— 命中"报错日志"+"这个报错怎么回事"
- "帮我看下这个 stack trace" —— 直接命中
- "我点保存按钮之后页面白了" —— 命中"异常表现"+"XX 坏了"
- "这个 bug 是从哪个 commit 引入的" —— 命中"这 bug 从哪来"
- "线上有个报警，日志在这里，帮我排查下" —— 命中"报错日志 / 排查异常"
- "我跑测试挂了，看下为什么" —— 命中"异常表现 / 排查"
- `/bug-hunter` —— 永远命中（手动调用不受 description 影响）

### 不应该命中（改完后）

- "帮我 review 下这个 PR" —— 被排除句明确挡住
- "我要写个登录功能" —— 没有故障信号
- "这段代码写得好吗 / 有没有更好的写法" —— code review 类，被排除
- "怎么在 Vue 里做 X" —— 没有故障信号，不是定位问题
- "加个错误处理" —— 是写防御性代码，不是排查已有 bug

### 可能的边界情况（需要后续 eval 观察）

- "这个函数有没有潜在 bug" —— **我的方案会不触发**（没有故障现象）。如果你希望这种也触发，得放松排除那句话。
- "单元测试跑红了" —— **会触发**（测试红算故障信号），这符合直觉。
- "性能变慢了" —— **边界模糊**。我倾向让它触发（"XX 不对"可以覆盖），但如果你有专门的 performance skill 就该在排除里加一句。

---

## 什么没改（§4.3 的最小化原则）

按 §3.2，用户的抱怨是 description，就只改 description。没碰正文、没碰文件组织、没加 `when_to_use` 字段（单条够用）、没动 frontmatter 别的字段。如果跑一段时间后发现正文里排查流程也有问题，再开第二轮优化。
