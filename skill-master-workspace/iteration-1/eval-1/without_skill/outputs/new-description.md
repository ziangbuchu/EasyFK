description: Root-causes a specific failing behavior from an error signal and pinpoints the offending code location. Use when the user presents a concrete failure artifact — an error message, exception, stack trace, crash log, test failure output, or a described symptom like "X throws / returns wrong value / hangs / crashes when Y" — and wants to know why it happens and where in the code to fix it. Works from the signal backward into the codebase: parses the trace, forms hypotheses, reads suspect files, and reports the root cause with file paths and line numbers. Skip for forward-direction work where no failure exists yet: writing new features, refactoring clean code, general code review, style or performance tuning, or "does this look right?" questions without a reported symptom. Also skip when the user has already identified the faulty line and only wants help writing the patch.
when_to_use: Trigger on concrete failure signals — pasted stack traces, error logs, exception names, "it crashes / throws / returns wrong X", failing test output, reproduction steps for a bug. Do not trigger on generic coding help, feature work, or reviews of code that is not reported as broken.
