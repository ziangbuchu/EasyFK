# process-notes

## 没问但假设了的事

用户的请求里有几处留白，我没去追问，按以下默认值推进，好让交付物一次到位：

1. **"自动"的强度**。用户说"每次写完代码之后自动给我一个小结"。严格意义上的"每次都自动"要靠 `settings.json` 里的 `Stop` hook 才能做到 — skill 的触发靠描述匹配，Claude 可能漏触发。我选了 skill 路线（因为用户明确要 skill），但在描述里写得很重："只要本轮有改动代码的动作就触发"，尽量贴近"自动"的体感。如果用户想要 100% 自动，下一步该配 hook。这点值得在交付时口头提一下。
2. **小结的粒度**。用户只说了三项："改了哪些文件 / 加了什么功能 / 有没有 breaking change"。我就按这三段写，不加测试、不加 TODO、不加性能分析——免得偏离用户的原始需求。
3. **语言**。用户用中文描述需求，默认小结输出中文。描述字段里中英文都放了，方便描述匹配（Claude Code 的 skill 匹配对语言不算敏感，但保险起见都放）。
4. **什么时候不触发**。我主动加了三条不触发条件（纯问答、极小改动、用户喊停），避免 skill 变成噪音源。这是基于常识，用户没说。
5. **要不要额外文件**。一开始考虑过给个 `template.md` 或 `examples/` 目录，但想了想觉得没必要——格式在 SKILL.md 里写死一份代码块示例足够了，多文件反而让用户维护成本变高。所以只交付一个 `SKILL.md`。

## description 怎么写的

skill 的触发靠 description。关键词覆盖：
- 动作触发词："写完代码 / 改完 / Edit / Write / MultiEdit / 结束一轮编辑"
- 产出词："中文小结 / 改动过的文件 / 功能 / breaking change"
- 范围词："每次 / 任何包含...的回合结尾"

中英双语是为了 description 匹配时两边的关键词都能命中——有些用户用英文提需求，有些环境里的 trigger 判断更吃英文。

## 结构上的取舍

SKILL.md 分了 5 个小节：何时触发 / 产出格式 / 怎么收集信息 / 写作准则 / 反例。

- **反例**这节是我刻意加的。用户说的是"让 AI 味消失"这种偏好（从用户记忆里看到过），光写"要简洁"没用，贴一个反例才治得住 Claude 的套话毛病。
- **和 git 的关系**最后一节是边界防御——不加这段，Claude 可能把用户在对话开始前就已存在的未提交改动也算进本轮小结，结果就错了。
- **没写**任何关于 commit message、PR、changelog 的内容——那是别的 skill 的活，避免功能蔓延。这也符合 CLAUDE.md 里"技能之间不要相互耦合"的项目约束。
