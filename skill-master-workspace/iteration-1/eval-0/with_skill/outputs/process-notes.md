# change-summary skill 的制作过程笔记

## 需求里被我按"合理默认"处理掉的不确定项（§2.1 本来应该问用户）

skill-master §2.1 要求钉死 5 件事再动手。这是测试环境，我没法真问用户，所以记下我默认了什么、为什么：

1. **skill 具体做什么**：写完代码后给一份中文交付小结。用户给得挺清楚，直接用。
2. **什么时候该触发**：用户没给"真实原话"。我从需求里抽了两类触发——显式求（"总结一下这次改动"、"刚才改了啥"）和 Claude 自己到交付节点。真跑这个 skill 的时候，description 里的触发语需要按用户实际说法再调一次。
3. **什么时候不该触发**：和 git commit message 场景很像，容易串。我在 description 和正文都显式划了边界。
4. **装哪一层**：用户点名项目级，照做。
5. **副作用**：只是生成文字、不改代码不推远端，没副作用，不用 `disable-model-invocation`。

## "自动触发" 这个需求我没能在 skill 层完全兑现——这是关键决策

用户说"Claude **每次写完代码之后自动**给我一个中文小结"。这里的"每次...自动"严格讲 skill 做不到：

- skill 的触发是 Claude 基于 description 语义判断的倾向性行为，不是哈尼斯层的硬保证
- 要做到"每次都触发"，正确工具是 `.claude/settings.json` 里的 Stop hook（或 PostToolUse hook），由哈尼斯在特定事件点强制执行
- 系统提示里列出来的 `update-config` skill 的 description 也明确说了："Automated behaviors ('from now on when X', 'each time X') require hooks configured in settings.json"

我做了两件事来对齐这个落差：

1. 在 skill 的 description 里加了"Claude 自己刚完成一批文件编辑、准备交付时主动调用"——把"自动性"尽量推到 description 的语义层，让 Claude 在交付节点更倾向于触发
2. 在 SKILL.md 末尾加了一节"关于'自动触发'"，老老实实告诉用户：skill 不保证每次都触发，要 100% 确定就去配 Stop hook，并指路 `update-config`

这比默默按需求写一个"假装能自动"的 skill 更诚实。skill-master §5 的陷阱列表里有一条"描述的是能力不是触发条件"——反过来我也不想让 description 骗人说自己能做它做不到的事。

## 结构选择：为什么没有 reference/ 也没有 scripts/

skill-master §2.2 说第一版尽量只有 SKILL.md，拆文件是为了给"正文塞不下但需要时能查到的"东西找地方。

这个 skill 的正文目前大概 80 行，远低于 400 行警戒线。内容全都是"每次调用都要用到的核心原则"——什么时候写、写哪四块、风格要求、和 commit 的边界。没有任何一段属于"只有 20% 调用会用到"的参考材料。所以不拆。

也没写 scripts/——生成小结这事完全是基于当前上下文的语言表达，不是确定性的数据处理，没有稳定逻辑可以脚本化。让 Claude 每次现写反而更贴合当下的改动。

## description 是这样写出来的（skill-master §2.3）

动词开头：`写完代码改动之后给用户一份中文小结`——直接讲动作。

然后拉触发语：

- `改了哪些文件、加/改/删了什么功能、有没有 breaking change`——原需求里的三个关键词都塞进来，因为用户未来在提起这事时很可能复用这些词
- `触发场景：用户说"总结一下这次改动"、"刚才改了啥"、"出个 changelog"、"这次动了什么"`——给了 4 条可能的用户原话，语义从正式到口语都覆盖一点
- `或者 Claude 自己刚完成一批文件编辑、准备交付时主动调用`——把自动性推到语义层

边界：

- `不要和 git commit message 生成混淆——这个是给人读的改动说明，不是 commit`——§2.3 和 §5 都强调了要明确边界，commit message 场景和这个高度重叠，必须显式排除

长度：看着差不多 200 字符，远低于 1536 上限，没有被截断风险。

## 正文风格的几个取舍（skill-master §2.4）

- **没用 MUST/NEVER**：§2.4 的硬要求。全篇改成"好：/差："或者"...就别写..."这种解释式语气。唯一一处稍微硬的是"这一节永远要写，哪怕是'没有'"——因为 breaking change 这一节跳过的话会真的误导用户，符合"跳过会真的导致错误"的例外条件。
- **带了正反例**：防抖那对 `给搜索框加了防抖 / 修改了 N 个文件`——skill-master 正文里也是这么教的，具体例子比抽象规则好记。
- **没穷举"如果 A 就 B，如果 C 就 D"**：§2.4 警告过这种列表 Claude 不会真按它跑。比如"文件数多时"、"重构时"、"只改测试时"这些分支我都没单独写，就一句"按改动规模酌情增减"的精神融在风格要求里。

## 我没做的自测（§2.6）

测试环境下没法真跑 Claude 看行为。如果真要上线，最少要跑三个场景：

1. 用户说"总结一下这次改动"——应该触发
2. 用户说"帮我写个 commit message"——**不该**触发这个 skill（应该走别的路径）
3. Claude 自己刚改完 3 个文件准备收尾——应该触发

第 2 条是这个 skill 最大的风险场景，上线后要盯一下。

## 交付格式

按 skill-master §2.7 的口径，交付时要告诉用户：装哪、怎么触发、副作用、下一步。我把这些信息都放在最终的 assistant 回复里（不是 SKILL.md 正文），因为 SKILL.md 是给 Claude 读的运行时指令，不是给用户看的 README。
