# change-summary

## 什么时候写这份小结

两种进入方式：

1. **用户显式要**——"总结一下"、"出个小结"、"这次改了啥"、"有没有 breaking"
2. **一次明确的交付节点刚结束**——你刚写完一个功能 / 修完一个 bug / 完成一轮重构，正要对用户说"改完了"。在说"改完了"之前，把这份小结先写出来，作为交付说明

如果只是中途改了一两个文件、还要继续做，不用写。小结是"告一段落"时才出的东西，不是每次 Edit 之后都刷。

## 要写什么

四块内容，按这个顺序，缺什么留空，别硬凑：

### 1. 这次做了什么（一句话）

用人话讲这次动作的目的，不是"修改了 N 个文件"。

- 好：`给搜索框加了防抖，输入停顿 300ms 才发请求`
- 差：`修改了 SearchBox.vue 和 useDebounce.ts`

### 2. 改了哪些文件

按"新增 / 修改 / 删除 / 重命名"分组。文件路径用仓库根的相对路径。每个文件后面跟一句说明它里面发生了什么——只写这个文件层面的事，不要重复整体目的。

```
修改：
- src/components/SearchBox.vue：把直接调 fetchResults 换成防抖包装
- src/composables/useDebounce.ts（新增）：通用防抖 composable

删除：
- src/utils/oldSearchHelper.ts：被 useDebounce 替代
```

### 3. 有没有 breaking change

这一节永远要写，哪怕是"没有"。让用户看完知道能不能直接合。

明确点出三类东西：

- **对外 API / 函数签名变了**（参数增删、返回值类型变、默认值变）
- **配置 / 环境变量 / 依赖改了**（需要用户动手的事）
- **行为语义变了**（同一个调用，以前这样现在那样——比如接口以前同步现在异步、默认开/关换了）

没有就写"无 breaking change"。有的话一条一条列出来，**每一条都说清楚受影响的调用方要怎么改**。

### 4. 还剩什么没做 / 用户接下来要干嘛（可选）

如果改动需要用户自己跑一下 migration、装新依赖、重启服务、补一段环境变量，就在这里点一句。没有就不写，别为了凑结构硬加。

## 风格要求

- **全中文**。代码标识符（函数名、文件路径、变量名）保持原样不翻译
- **短**。整份小结默认 10-25 行量级。文件多了可以长一些，但不要每个文件都写三句话
- **说"什么变了"，不说"我改了"**。用户关心的是结果不是过程
- **不要总结你没做的事**。如果没跑测试就别写"测试通过"，如果没读过某个文件就别在小结里提它
- **别复述 diff**。这份东西是人读的摘要，不是 patch

## 和 git commit 的区别

commit message 是给版本历史看的，要机器可解析、通常一行标题加 body。这份小结是给正在看屏幕的用户看的一次性交付说明，结构更松、信息量更大、可以用 Markdown。两者不要混用——如果用户明确要 commit message，走那条路，别用这份模板。

## 关于"自动触发"

这个 skill 的 description 让 Claude 在"完成一轮改动后"倾向于主动写小结，但实际是否触发取决于 Claude 在那一刻对场景的判断，不是硬保证。

如果你（用户）希望**每次**代码写完都无条件出一份小结——skill 本身做不到这件事，需要在 `.claude/settings.json` 里配一个 Stop hook 来兜底。这是项目哈尼斯层的配置，不是 skill 层能覆盖的。可以让 Claude 用 `update-config` skill 帮你加。
