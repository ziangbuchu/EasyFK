# skill-master description 精简提案

## 当前版本（原文）

> 做 skill（Claude Code 技能）相关的工作时进入——从零写一个新 skill、调优已有 skill、把反复使用的流程沉淀成 skill、或者在 skill 不触发 / 乱触发时修 description。也能读自己的文件优化自己。用户说"帮我写个 skill / 做个技能 / 优化 skill / 调 description / 这个 skill 不触发 / skill-master 优化下自己 / 把这段流程做成 skill"时都进来。

三句话、约 140 字。问题在第三句：它把前两句的四个场景用"用户原话"重新列了一遍，信息基本重复。

## 提案版本（新 description）

> 做 skill（Claude Code 技能）相关的工作时进入——写新 skill、把反复用的流程沉淀成 skill、调优已有 skill 的行为或内容、修不触发 / 乱触发的 description，也能读自己的文件优化自己。

一句话、约 80 字。

## 保留了什么

- **"skill（Claude Code 技能）"的双语锚点**：英文关键词和中文"技能"两种说法用户都可能说，保留让两种都能命中。
- **四个核心触发场景全部保留**：
  1. 写新 skill（= "从零写一个新 skill"）
  2. 把反复用的流程沉淀成 skill（= 原"把反复使用的流程沉淀成 skill / 把这段流程做成 skill"）
  3. 调优已有 skill（= "优化 skill"）
  4. 修 description 不触发 / 乱触发
- **自我优化场景**：保留"也能读自己的文件优化自己"这句，因为它是一个相对特殊、不那么 obvious 的入口，值得单独点名。
- **"进入"开头的触发动词**：保持触发语气，让模型读到就知道这是一条匹配规则。

## 删了什么、为什么

- **删掉整个"用户说 XXX 时都进来"的例句清单**。原因：
  - 这些例句（"帮我写个 skill / 做个技能 / 优化 skill / 调 description / 这个 skill 不触发 / skill-master 优化下自己 / 把这段流程做成 skill"）和前面的场景描述是一一对应的，本质是同一信息换个表达方式。
  - 现代模型做触发匹配时，"写新 skill"和用户说的"帮我写个 skill / 做个技能"之间语义跳转是零成本的，不需要把同义词都堆到 description 里。
  - description 每个字都一直在上下文里占位，重复信息是纯浪费。
- **把"从零写一个新 skill"压成"写新 skill"**。"从零"是冗余——写新的当然是从零。
- **把"skill 不触发 / 乱触发时修 description"调整为"修不触发 / 乱触发的 description"**。同样意思，少几个字。
- **合并"调优已有 skill"和"修 description"里的重复主体**。原文两处都重复了"skill"，新版把调优和修 description 并列在同一句里，语义不丢。

## 效果

- 字数从约 140 字降到约 80 字，上下文占用接近减半。
- 四个触发场景一个不少，自我优化这个相对隐蔽的入口也保留。
- 没有例句列表导致的"触发词过拟合"风险降低——原来那一串加引号的原话，模型容易以为必须命中才触发。
