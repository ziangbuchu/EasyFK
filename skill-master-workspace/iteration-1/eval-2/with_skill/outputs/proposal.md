# skill-master description 精简提案

## 背景

用户反馈当前 description "读着有点长"。不是字符超限问题（当前 228 字符，远未到 1536 上限），是**可读性**问题——同样的意思用两种形式各说一遍（能力列表 + 触发语列表），信息密度低。

## 当前 description

```
做 skill（Claude Code 技能）相关的工作时进入——从零写一个新 skill、调优已有 skill、把反复使用的流程沉淀成 skill、或者在 skill 不触发 / 乱触发时修 description。也能读自己的文件优化自己。用户说"帮我写个 skill / 做个技能 / 优化 skill / 调 description / 这个 skill 不触发 / skill-master 优化下自己 / 把这段流程做成 skill"时都进来。
```

228 字符。

## 冗余诊断

把"能力描述"和"触发语样例"两段对照一下，大半是重复的：

| 能力描述段说的 | 触发语段说的 | 重复度 |
|---|---|---|
| 从零写一个新 skill | "帮我写个 skill / 做个技能" | 完全重复 |
| 调优已有 skill | "优化 skill" | 完全重复 |
| 把反复使用的流程沉淀成 skill | "把这段流程做成 skill" | 完全重复 |
| 在 skill 不触发 / 乱触发时修 description | "调 description / 这个 skill 不触发" | 完全重复 |
| 也能读自己的文件优化自己 | "skill-master 优化下自己" | 完全重复 |

按 techniques.md §1.1 的结构（做什么 + 触发条件 + 原话样例 + 边界），当前文本相当于把"做什么"和"原话样例"各写了一遍，中间还用自然语言又铺排了一次。

## 提案新 description

```
做 skill（Claude Code 技能）相关的事时进入——新建、调优、把反复流程沉淀成 skill、修触发不对的 description，或者读自己的文件优化 skill-master。典型原话："帮我写个 skill"、"这个 skill 不触发 / 乱触发"、"把这段流程做成技能"、"skill-master 优化下自己"。
```

大约 130 字符，比原来短约 43%。

## 保留了什么、丢了什么

### 保留（每个都是不可妥协的触发场景）

- **新建 skill**："新建" + 触发语 "帮我写个 skill"
- **调优已有 skill**："调优" + 触发语 "这个 skill 不触发 / 乱触发"（覆盖 description 问题 + 行为问题两种子情况）
- **流程沉淀**："把反复流程沉淀成 skill" + 触发语 "把这段流程做成技能"（这个触发语场景用户很常说，独立保留）
- **description 修 bug**："修触发不对的 description"（这是最常见的优化子类，单独保留有助于 Claude 在"skill 不触发"这种含糊说法下仍命中）
- **自我优化**："读自己的文件优化 skill-master" + 触发语 "skill-master 优化下自己"（skill-master 特有路径，必须点名）

### 丢弃（冗余或低价值）

- 能力段和触发语段的镜像重复——保留一次即可
- 触发语里的 "做个技能" ≈ "帮我写个 skill"，合并
- 触发语里的 "优化 skill" ≈ "这个 skill 不触发 / 乱触发"（前者太泛，后者更具体，留后者）
- 触发语里的 "调 description" ≈ "这个 skill 不触发"（后者覆盖前者的场景，留后者）
- "也能...也..." 这类连接词

### 不改动的东西

- 正文（§0-§6 全部）——用户只抱怨 description，按 §4.3 "别扩大 scope" 原则保持最小改动
- `reference/anatomy.md` 和 `reference/techniques.md`

## 可选变体：拆 when_to_use

如果想让 description 本体再短一点，按 techniques.md §1.4 的写法可以拆：

```yaml
description: 做 skill（Claude Code 技能）相关的事时进入——新建、调优、把反复流程沉淀成 skill、或者让 skill-master 优化自己。
when_to_use: 典型原话："帮我写个 skill"、"这个 skill 不触发 / 乱触发"、"把这段流程做成技能"、"skill-master 优化下自己"。
```

拼起来总长度差不多，但 description 本身更轻。**不强推**——单字段也完全够用，拆字段只在 description 本身要参与别处展示（比如 `/` 菜单摘要）时更划算。推荐先采用单字段版。

## 自我优化的几条 caveat

按 §4.1 / §4.2 明确告知：

1. **当前会话不生效**。SKILL.md 一次会话只加载一次。提案被接受、文件改完之后，当前这轮对话里 Claude 看到的还是旧 description；要下次会话才会用新的。
2. **触发率需要实地验证**。description 的好坏只有放回真实用户消息流里才知道。建议改完之后刻意说几句：一条该触发的模糊说法（比如 "这个流程我想固化下"）、一条不该触发的相邻说法（比如 "帮我改下这段代码"），看命中是否稳定。
3. **按 §4.3 "别扩大 scope"**。这次只动 description。正文里如果你觉得还有问题（比如某段话啰嗦、某个例子不准），开下一轮再说，不要混进这次改动里。
4. **保留的触发语全部是用户原话**。如果你的实际说法和这里不一样，告诉我，以你的原话为准——description 原话命中率永远高于推断的同义词。
